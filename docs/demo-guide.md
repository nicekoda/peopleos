This is the practical "how to run a demo" companion to
`docs/security.md`/`docs/architecture.md` — it doesn't restate the RBAC
design or the tenant-isolation model, only how to log in, what to click,
and what each role should (and shouldn't) be able to do. All data below
lives on the `uesl` tenant, seeded by `DemoDataSeeder` (see
`database/seeders/DemoDataSeeder.php`).

## 1. Local Setup Reminder

See `README.md` → "Local Development Setup" for the full first-time
setup. Once the project is installed and `.env`/`SEED_USER_PASSWORD` are
configured, get a fresh demo database with:

```bash
./artisan.bat migrate:fresh --seed
```

> **⚠ Destructive command — local/demo environments only.**
> `migrate:fresh` drops every table and rebuilds the schema from
> scratch before seeding — **anything currently in that database is
> gone**, with no confirmation prompt and no recovery. Only ever run
> this against your own local development database
> (`DB_DATABASE=peopleos_dev` by default). **Never run this — or any
> `migrate:fresh` — against a production database.** See
> `docs/production-readiness.md` "Admin & Demo Accounts" for the
> production rule this backs: production seeding, if ever needed, runs
> `TenantSeeder`/`PermissionSeeder`/`RoleSeeder` individually, never the
> full `DatabaseSeeder` chain this command triggers.

This runs `TenantSeeder` → `PermissionSeeder` → `RoleSeeder` →
`UserSeeder` → `DemoDataSeeder`, in that order. `DemoDataSeeder` only
seeds the `uesl` tenant — `airpeace`/`ibom` keep their existing minimal
Tenant Admin-only setup, so the tenant count never grows past the three
already established.

## 3. Suggested Demo Login Sequence

A natural order to show the app, each login building on the last:

1. **Tenant Admin** — company profile, full Users & Access, full Settings.
2. **HR Manager** — day-to-day operations: employees, leave, documents, policies.
3. **HR Officer** — the same modules with a narrower lens (no user/role management).
4. **Line Manager** — approves Chidi Okafor's pending leave request; is blocked from approving anyone outside their reporting line.
5. **Employee** (Chidi Okafor) — self-service only: own profile, own leave request, own documents.
6. **Auditor** — audit log, read-only everywhere else.
7. **Platform Super Admin** — logged in at the base domain, to show the deliberately different (safe, non-tenant) experience.

## 5. What Each Role Should Be Able to See

- **Tenant Admin** — everything within the tenant: all modules, all Settings sections, full Users & Access, plus the only role that can enable/disable modules (Settings → Modules) — branding management is shared with HR Director/HR Manager, module toggling is not.
- **HR Manager** — employees, leave (tenant-wide), documents, policies, Onboarding/Offboarding (full manage, tenant-wide), HR Document Templates & generated documents (full manage), Recruitment (full manage — job openings and applications, including delete/mark-ready-for-conversion and converting a hired application to an employee), Settings (Document Categories/Leave Types/Departments/Positions/Locations/HR Document Templates, full CRUD), branding (view + manage), modules (view only — cannot toggle), custom fields (view only — cannot create/edit/disable fields), but not Users & Access or Audit Log.
- **HR Officer** — leave (tenant-wide) and policies, Onboarding/Offboarding (create/update/assign/complete, no delete), HR document templates (view only) and generated documents (view/create/generate/update/submit, no delete, and — deliberately — no approve/reject), Recruitment (view/create/update/manage applications/add notes/mark ready for conversion, no delete, and — deliberately — no candidate-to-employee conversion), Settings visible (for Leave Types/Departments/Positions/Locations — view/create/update, no delete), but not Users & Access, Document Categories management, or Audit Log.
- **Line Manager** — their own profile plus direct reports only; leave approval limited to direct reports; sees and can complete Onboarding/Offboarding tasks only for direct reports or assigned to them; no Settings access; no HR document access (deliberately none by default); no Recruitment access (deliberately none by default — no assigned-interviewer scoping model exists yet).
- **Employee** — their own profile, own leave, own documents, own policy acknowledgements, and only Onboarding/Offboarding tasks assigned to them personally. No Settings, no visibility into other employees' data, no HR document access (deliberately none by default), no Recruitment access (deliberately none by default).
- **Auditor** — tenant-wide read access to leave/employees/Onboarding/Offboarding (view-only) and the Audit Log, plus view-only access to HR document templates/generated documents and Recruitment (job openings/applications); Settings visible (for Security & Audit) but no admin write actions anywhere.
- **Platform Super Admin** — a safe, empty-of-tenant-data dashboard; blocked (403) from every tenant-scoped `/api/v1` endpoint, including its own tenant's settings.

None of the above is enforced by hiding navigation links — every rule is
re-checked server-side on every request. See `docs/security.md` for why
the frontend is never the security boundary in this app.

## 6. Known Limitations

- No invitation flow, MFA, or SSO — demo users are pre-seeded, not self-registered. A real password reset flow exists (Checkpoint 44 — "Forgot your password?" on the login page), though demo users shouldn't need it since credentials are already documented above.
- No RBAC role/permission *editing* UI yet — roles/permissions are viewable, not editable, from the UI.
- No payroll, performance, notifications, exports, or analytics charts. Onboarding/Offboarding (Checkpoint 33) exists as a foundation — no task templates, approval routing, or notifications yet.
- HR Documents & Letter Generation (Checkpoint 34) has no DOCX file, e-signature, automated sending, bulk generation, or employee self-service download. PDF download was added in Checkpoint 35 — generated on demand, never stored on disk, with a plain-text "not yet approved" banner on any non-approved document (Checkpoint 37). Template version history was added in Checkpoint 36 — no diff/compare view between versions. A single-approver approval workflow was added in Checkpoint 37 — no multi-level approval routing and no notifications when a document changes state. 8 starter templates and a Duplicate action were added in Checkpoint 38 — tenant-specific only (seeded for `uesl` only, not a global/shared library, and not automatically seeded for new tenants), no AI generation.
- Recruitment & Applicant Tracking (Checkpoint 39) is a foundation only — job openings, applicants/applications, a pipeline stage, and internal notes are real, and candidate-to-employee conversion was added in Checkpoint 40 (a real `Employee` row is created, gated by its own permission), but there's still no automatic `User` account/role assignment/onboarding start on conversion, no public candidate portal, no CV parsing/AI screening, no interview scheduling, no offer approval/automation, and no bulk import/conversion. No demo data is pre-seeded for this module — create job openings/applications live during a demo.
- No platform-level dashboard for the Platform Super Admin (deliberately kept minimal/safe instead).
- Module Registry & Branding (Checkpoint 47) is enablement only — no subscription/entitlement layer, so every tenant can toggle any of the six toggleable modules regardless of what they'd commercially be "subscribed" to in a future billing model (see `docs/platform-vision.md`). No login-page or email branding — only the in-app UI picks up the tenant's logo/colors. No max-dimension warning beyond outright rejection of an oversized logo.
- Custom Fields support Recruitment Applicants (Checkpoint 48), Job Applications (Checkpoint 49), and Employees (Checkpoint 51) only — job openings, leave, onboarding/offboarding, and every other entity have no custom-field support yet (documented roadmap: `lifecycle_processes`/`leave_requests` next). Adding Job Applications and Employees as the second and third entities each required no schema or core-service change — proof the engine generalizes. Field-level access is enforced (Checkpoint 50) via three fixed permissions (`custom_fields.access_sensitive`/`access_confidential`/`access_restricted`) — Tenant Admin holds all three, HR Manager holds `access_sensitive` only, HR Director holds none by default (an intentional MVP decision, not an oversight); a tenant-configurable, role-based visibility-rules override exists as of Checkpoint 53 (`can_view`/`can_edit` per field per role, Tenant Admin role cannot be targeted, no per-user rules, no conditions/expressions). Custom Forms (Checkpoint 52) exist for Employee only — a form groups existing custom fields into sections for display, but never a second way to read or write a value; there is no drag-and-drop designer, only simple up/down reordering, and no workflow conditions, reports, or dashboards consuming form structure yet. `is_required_override` on a form field is display-only — it is never enforced server-side, only the underlying custom field's own required flag is. Converting an application to an employee never copies its custom field values — Employee custom fields are entirely separate, employee-scoped values. As of Checkpoint 54, the Employee Show page no longer renders every viewable custom field in both the form and the fallback card — a field assigned to an active form/section/field row renders only in the form, and the fallback card shows only unassigned fields; this cleanup is Employee-only, Recruitment's `ApplicationShow.tsx` still shows both its custom-fields cards unconditionally (it has no Custom Forms yet). Assigning the same custom field to two different active forms is still allowed and unrestricted — it renders in both, by design.
- Leave balances have no accrual engine or carry-forward automation — the seeded balances are a fixed, consistent snapshot, not a running calculation.
- Documents use safe fake files on the private `local` disk — there is nothing to actually "view" as a real PDF; the demo shows metadata (title, category, sensitivity, expiry), not real document contents.
- See `docs/security.md` → "Known Limitations / Follow-up" for the complete, checkpoint-by-checkpoint list.

## 7. What Not to Demo Yet

Don't attempt to show, or promise as working: full RBAC permission editing, self-service invitation/registration, MFA/SSO, payroll, performance modules, AI features, notifications, data exports, analytics dashboards, a platform-wide admin dashboard, billing/subscription/entitlement management (module toggling exists — a tenant subscribing to only *some* modules commercially does not), custom fields on any entity other than Recruitment Applicants/Job Applications/Employees, custom forms on any entity other than Employees, a drag-and-drop form designer, a form-assignment wizard, conditional sections, form versioning, per-user or condition-based field-visibility rules (only role-based `can_view`/`can_edit` overrides exist, Checkpoint 53), report/export/AI-filter enforcement of visibility rules, a workflow/rules/approval builder, a page/layout designer, custom reports/dashboards using custom fields, application-field-to-employee-field mapping on conversion, third-party integrations, a workflow/approval-routing builder for Onboarding/Offboarding, task templates, DOCX generation or e-signature for HR Documents, multi-level/routing approval for HR Documents, bulk HR letter generation, a global/shared HR document template library across tenants, template import/export, template ratings, a public candidate portal, job-board posting, CV parsing/AI screening, interview scheduling, offer approval/automation, automatic user-account/role-assignment/onboarding-start on candidate conversion, or bulk conversion. PDF download (Checkpoint 35), the single-approver approval workflow (Checkpoint 37), the seeded starter templates + Duplicate action (Checkpoint 38) for HR Documents, the Recruitment & Applicant Tracking foundation — job openings, applications, pipeline stages, internal notes, and the "ready for conversion" milestone flag (Checkpoint 39) — candidate-to-employee conversion into a real Employee record (Checkpoint 40), per-tenant module enable/disable + branding (logo/colors, Checkpoint 47), custom fields for Recruitment Applicants (Checkpoint 48), Job Applications (Checkpoint 49), and Employees (Checkpoint 51), field-level visibility/sensitive field access for custom fields (Checkpoint 50), a Custom Forms foundation for Employees (Checkpoint 52), configurable, role-based field visibility rules (Checkpoint 53), and the form-aware Employee custom-fields UI cleanup (Checkpoint 54) *are* real. If asked about any of the others, the honest answer is "not built yet — see the roadmap in `docs/architecture.md` and the long-term direction in `docs/platform-vision.md`."
